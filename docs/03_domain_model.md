# 03_DOMAIN_MODEL（领域模型）

> 本文档用于定义本系统中**最核心的概念对象（Domain）**及它们之间的关系。
> 目标是：
> - 为数据结构设计提供稳定锚点
> - 为功能设计提供统一语言
> - 防止后期开发中概念混乱、语义漂移

---

## 一、领域模型设计原则

在开始具体定义前，先明确本系统的领域建模原则：

1. **以“人的长期记忆与体验”为中心，而非以功能为中心**
2. **概念必须贴近自然语言，而非技术语言**
3. **优先抽象“可复用的概念”，而不是具体实现**
4. **允许不完美、不完整、模糊的记录存在**（这是本系统的核心价值之一）

---

## 二、核心领域对象（Core Domains）

当前阶段（MVP + 中期扩展）的核心领域对象包括：

1. Item（物品）
2. Location（位置）
3. Experience（体验 / 使用记录）
4. Feeling（感受 / 情绪）
5. Event（事件）
6. Tag（标签）

以下逐一说明。

---

### 1️⃣ Item（物品）

**定义**：
> 现实生活中可被管理、查找、使用、消耗或评价的“具体事物”。

**典型示例**：
- 家里的某件衣服
- 某个收纳盒
- 一杯具体品牌 + 口味的奶茶

**Item 的核心特征**：
- 是「被管理对象」
- 可以被多次使用
- 可以关联位置、体验、感受

**关键属性（概念层）**：
- 标识信息（名称、别名）
- 分类信息（衣物 / 食品 / 日用品等）
- 当前状态（在用 / 闲置 / 已消耗 / 已丢弃）

> ⚠️ 注意：
> Item 本身 **不等于评价、不等于情绪**，它只是一个“承载体”。

---

### 2️⃣ Location（位置）

**定义**：
> 对物品或事件发生地点的抽象描述。

**典型示例**：
- 家 → 卧室 → 衣柜 → 左下角
- 冰箱 → 冷藏室 → 第二层

**设计要点**：
- 位置是 **层级结构**
- 不要求精确，只要“对用户有意义”

**核心能力**：
- 用于「找回物品」
- 用于「减少重复购买」

---

### 3️⃣ Experience（体验 / 使用记录）

**定义**：
> 某一次用户与 Item 或 Event 的具体互动记录。

这是本系统中**非常重要的中间层对象**。

**典型示例**：
- 第一次喝某款奶茶
- 穿某件衣服出门一天
- 使用某个收纳方式一段时间

**Experience 的特点**：
- 是「一次性的」
- 有时间属性
- 是 Feeling 的主要载体

---

### 4️⃣ Feeling（感受 / 情绪）

**定义**：
> 用户在某个体验或事件中产生的主观感受。

**重要声明**：
- Feeling 是 **允许模糊、矛盾、未命名的**
- 不强制情绪分类体系

**示例**：
- “有点腻”
- “当时很委屈，但现在说不清原因”
- “这杯点无糖会更好”

> Feeling 是这个系统中，**最私人、最不追求结构化的对象**。

---

### 5️⃣ Event（事件）

**定义**：
> 对用户产生影响的一次现实发生的事情。

**示例**：
- 与某人的一次对话
- 被一句话刺到
- 一次失败的消费决策

**与 Experience 的区别**：
- Event 偏“发生了什么”
- Experience 偏“用户如何经历了它”

---

### 6️⃣ Tag（标签）

**定义**：
> 用于连接、归类、回溯的轻量级标记。

**设计原则**：
- 标签是自由的
- 不追求体系完整
- 可用于 Item / Experience / Feeling / Event

---

## 三、领域对象之间的关系（简述）

以自然语言描述核心关系：

- 一个 **Item** 可以：
  - 位于一个或多个 Location（随时间变化）
  - 关联多条 Experience

- 一个 **Experience**：
  - 通常关联一个 Item 或一个 Event
  - 可以包含 0~多条 Feeling

- 一个 **Event**：
  - 可以触发 Experience
  - 可以直接关联 Feeling

- Feeling 永远 **依附于 Experience 或 Event 存在**

---

## 四、当前阶段刻意不建模的内容（Out of Scope）

以下内容 **暂不进入领域模型**：

- 用户系统（默认单用户）
- 社交 / 分享
- 情绪分析算法
- 自动推荐与决策系统

---

## 五、阶段性总结

这一领域模型的核心目的不是“完美抽象”，而是：

> **让用户未来能准确回忆：
> 当时发生了什么、用户怎么感受的、为什么会踩雷。**

当实现与设计发生冲突时：
- 优先保证「体验与感受可被留存」
- 而不是保证数据结构的优雅
