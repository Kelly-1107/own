# 04_FUNCTIONAL_MODULES（功能模块设计）

> 本文档用于描述系统在当前与可预见扩展阶段中，应当具备的**功能模块划分**。
>
> 本文档关注的是：
> - 系统“能做什么 / 不做什么”
> - 模块之间的职责边界
>
> **不涉及：**
> - 页面结构
> - 技术实现
> - 具体 API 或数据表

---

## 一、模块设计总原则

1. **功能围绕“回忆 + 决策 +掌控感”展开**
2. **每个模块都必须能独立成立，而不是互相绑死**
3. **允许用户以最低成本记录，而非最高质量记录**
4. **系统不替用户做判断，只负责“记住与呈现”**

---

## 二、模块总览

当前阶段模块划分如下：

1. Item 管理模块（物品）
2. Location 管理模块（位置）
3. Experience 记录模块（体验）
4. Feeling 记录模块（感受）
5. Event 记录模块（事件）
6. 搜索 / 回溯模块
7. 标签与关联模块

---

## 三、各模块职责说明

### 1️⃣ Item 管理模块

**核心职责**：
- 创建、编辑、查看 Item
- 管理 Item 的基本状态（在用 / 闲置 / 已消耗）

**允许的行为**：
- 快速创建 Item（信息可极简）
- 关联 Location
- 查看与 Item 相关的 Experience 记录

**刻意不做的事**：
- 不要求完整属性
- 不强制评分或评价

---

### 2️⃣ Location 管理模块

**核心职责**：
- 管理对用户有意义的“位置描述”

**允许的行为**：
- 层级化位置创建
- 模糊描述（如“柜子最里面”）

**刻意不做的事**：
- 不追求物理精确
- 不自动推断或纠错

---

### 3️⃣ Experience 记录模块

**核心职责**：
- 记录用户与 Item / Event 的一次具体互动

**允许的行为**：
- 轻量快速记录（时间 + 简述）
- 关联 Item 或 Event

**刻意不做的事**：
- 不强制填写 Feeling
- 不要求完整描述

---

### 4️⃣ Feeling 记录模块

**核心职责**：
- 留存用户当下或事后产生的主观感受

**允许的行为**：
- 文字自由输入
- 可事后补写
- 可附着于 Experience 或 Event

**刻意不做的事**：
- 不强制情绪分类
- 不做情绪分析与建议

---

### 5️⃣ Event 记录模块

**核心职责**：
- 记录对用户产生影响的一次“事情”

**允许的行为**：
- 单独记录事件
- 关联 Feeling

**刻意不做的事**：
- 不追求事件客观还原
- 不判断对错

---

### 6️⃣ 搜索 / 回溯模块

**核心职责**：
- 帮助用户找回过去的信息

**允许的行为**：
- 按关键词搜索
- 按时间线回溯
- 从 Item / Event 进入相关 Experience

**刻意不做的事**：
- 不做智能推荐
- 不做自动总结

---

### 7️⃣ 标签与关联模块

**核心职责**：
- 建立跨对象的轻量连接

**允许的行为**：
- 自由创建标签
- 标签复用

**刻意不做的事**：
- 不强制标签规范
- 不自动合并或建议

---

## 四、模块之间的协作关系（简述）

- Item / Event 是“入口对象”
- Experience 是中间承载层
- Feeling 是最底层、最自由的记录

系统应始终支持：
> **从任意一个点，慢慢找回完整上下文**

---

## 五、阶段性边界声明

在当前阶段，系统明确**不承担以下职责**：

- 替用户做消费决策
- 帮用户“想清楚”情绪
- 给用户提供人生建议

系统存在的意义是：
> **当用户需要的时候，它记得；
> 当用户不想面对的时候，它安静。**
